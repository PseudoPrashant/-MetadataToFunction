What does random_state=42 do?
Without it, the split is random every run — you'd get different rows each time. Setting random_state=42 freezes the randomness so the split is identical every time you run the code. The number 42 is just a popular convention — any number works.

What is random_state doing?
When train_test_split runs, it randomly shuffles your 1080 rows before splitting. That randomness comes from a number called a seed.

Why does the seed matter?
Run this in your head — same code, no random_state:
Run 1: Training rows → [row5, row2, row900 ...]  Accuracy: 97%
Run 2: Training rows → [row1, row88, row340 ...] Accuracy: 94%
Run 3: Training rows → [row7, row600, row12 ...]  Accuracy: 98%
Your accuracy changes every run — which makes it impossible to know if your model actually improved or if you just got lucky with the split.
With random_state=42:
Run 1: same rows → Accuracy: 97%
Run 2: same rows → Accuracy: 97%
Run 3: same rows → Accuracy: 97%
Consistent and reproducible every time. 


Why specifically 42?
Here's the honest answer — there is no technical reason whatsoever.
42 became a joke in the ML/data science community. It's a reference to the book "The Hitchhiker's Guide to the Galaxy" by Douglas Adams, where a supercomputer is asked:
"What is the answer to life, the universe, and everything?"
After millions of years of computing, it answers:
"42"
Programmers found it funny and started using it as their default seed. It spread across tutorials, textbooks and courses until it became an unofficial global convention.

TF-IDF Explained Simply
You have this sentence:
"function to add two integers and returns their sum"
The model can't read words — it only understands numbers. So we need to convert every word into a number that represents how important that word is.
But not all words are equally useful:
"function"  → appears in EVERY row → tells us nothing useful
"add"       → appears in only a few rows → very helpful!
"to"        → appears everywhere → useless
"integers"  → fairly specific → somewhat helpful
TF-IDF captures exactly this idea.



Part 1 — TF (Term Frequency)
How often does this word appear in THIS row?
Row: "add two integers add sum add"
      
count("add")      = 3
count("integers") = 1
count("two")      = 1
total words       = 6

TF("add")      = 3/6 = 0.50
TF("integers") = 1/6 = 0.17
TF("two")      = 1/6 = 0.17
More occurrences in this row → higher TF score.



Part 2 — IDF (Inverse Document Frequency)
How rare is this word across ALL rows?
Imagine you have 1080 rows total:
"function"  → appears in all 1080 rows
"add"       → appears in only 12 rows
"celsius"   → appears in only 4 rows

IDF("function") = log(1080/1080) = log(1) = 0.0  → useless word
IDF("add")      = log(1080/12)   = log(90) = 4.5  → useful word
IDF("celsius")  = log(1080/4)    = log(270)= 5.6  → very useful word
Rarer word across all rows → higher IDF score.



Final score — TF × IDF
TF-IDF("function") = 0.17 × 0.0 = 0.00  ← ignored
TF-IDF("add")      = 0.50 × 4.5 = 2.25  ← important
TF-IDF("celsius")  = 0.17 × 5.6 = 0.95  ← important

TF-IDF gives each word a score — high score if the word appears often in THIS row but rarely across ALL rows. Common words like "function" score zero. Unique words like "celsius" score high. The result is a row of numbers that represents the meaning of your text.



Multinomial Naive Bayes Algorithm
During Prediction — it combines probabilities
A new input comes in:
"adds two integers int a int b MathUtils add sum arithmetic 2"
For each possible function name it calculates:
P(addNumbers | "add","integers","sum","arithmetic")
  = P(add|addNumbers) × P(integers|addNumbers) × P(sum|addNumbers) × ...
  = 0.96 × 0.91 × 0.88 × 0.85
  = 0.65  ← highest ✅

P(subtractNumbers | "add","integers","sum","arithmetic")
  = 0.04 × 0.80 × 0.10 × 0.05
  = 0.00016

P(readFile | "add","integers","sum","arithmetic")
  = 0.01 × 0.02 × 0.01 × 0.01
  = 0.000000002
It picks the function name with the highest probability:
→ addNumbers ✅

Why is it called "Naive"?
It makes one naive assumption — it treats every word as completely independent from every other word.

Is that a problem?
Surprisingly — not really, especially for your use case:
Real world accuracy with Naive Bayes:  98.61% ✅
Even though the assumption is technically wrong, the algorithm still works extremely well for text classification. That's why it's been used for decades — spam filters, sentiment analysis

Why "Multinomial" Naive Bayes specifically?
There are different flavours of Naive Bayes:
Type                    Best for
MultinomialNB           text with word counts and TF-IDF scores
GaussianNB              continuous numbers like height, weight
BernoulliNB             binary data — word present or not
Since TF-IDF gives us count-based scores for words, MultinomialNB is the perfect match.

One line summary

Naive Bayes learns the probability of each function name given each word during training. At prediction time it multiplies those probabilities together and picks the function name with the highest score.

What is Inference?
Training = teaching the model
Inference = using the model to predict
Think of it like this:
Training   → student studies for weeks    → happens once
Inference  → student takes the exam       → happens anytime
You never retrain — you just load the saved model and ask it questions.


Why only pickle this time?
During training you needed many libraries because you were building and teaching the model. During inference you just:

Load the already trained model
Feed it an input
Get a prediction back

The model already has TF-IDF and Naive Bayes baked inside it — you don't need to import them again.
